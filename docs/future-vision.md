# Future Vision

## Important boundary

These are future possibilities, not MVP commitments.

They only matter after Pullstart proves that contract-driven onboarding is useful on a real repo.

## Possible later directions

- richer recovery and resume behavior after failed setup attempts
- contract generation from repo evidence
- learned failure patterns across repeated onboarding runs
- portability across repo families instead of one proof repo type
- deeper verification beyond "runnable enough"
- optional runtime helpers for repeatable local environments

## What stays separate from Pullstart MVP

Even if related ideas become interesting later, Pullstart should not start life as:

- a broad platform for every developer workflow
- a broad workflow orchestration product
- a project memory product
- a developer control plane

Those are adjacent spaces, not the first product.
